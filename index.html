展示组件和容器组件

发表于 2017-08-20 | 分类于 翻译 | | 阅读次数:
字数统计: 1,640

原文《Presentational and Container Components》by Dan Abramov

在编写React应用时，我发现了一个简单却非常有用的模式。如果你已经进行了一段时间的React开发，很可能已经发现了这个模式。这篇文章将这个模式讲得很清楚，但是我想要再补充几点。

如果把组件拆分成两类，你会发现这些组件能够更容易地被复用和描述。我将其称为容器组件(Container Components)和展示组件(Presentational Components)，我也听过诸如Fat和Skinny, Smart和Dumb, Stateful和Pure, Screens和Components的说法。这些分类方法并不完全相同的，但是他们的核心思想是相似的。

我的展示组件是这样的：

  • 关心事物看上去是什么样的。
  • 里面可能包含展示和容器类型的组件，且通常会有一些自己的DOM标记和样式。
  • 一般能通过this.props.children进行控制。
  • 不依赖应用的其他部分（比如Flux action和store）。
  • 不指定数据如何加载或操作。
  • 只通过props获取数据和回调。
  • 很少拥有自己的state（如果有，一般是描述UI的state而不是数据）。
  • 除了需要使用state，生命周期钩子（lifecircle hooks）或性能优化的情况外，一般会写成函数组件（Functional Components）。
  • 比如：Page, Sidebar, Story, UserInfo, List。

我的容器组件是这样的：

  • 关心事物是怎么运作的。
  • 可能包含展示组件和容器组件，一般组件本身不会有任何除了用来包裹的div以外的DOM标记，且绝对不会有任何样式。
  • 为展示组件和容器组件提供数据和行为。
  • 调用Flux actions并将这些调用作为回调传给展示组件。
  • 通常有作为数据源使用的状态。
  • 一般由高阶组件（HOC，如React Redux中的connect()，Relay中的createContainer()或Flux Utils中的Container.create()等）生成而不是手动编写。
  • 例如：UserPage, FollowerSidebar, StoryContainer, FollowedUserList。

为了使得二者的区分更清晰，我会将他们放在不同的文件夹中。

这样做的好处

  • 关注点分离。这种写组件的方式可以让你更好地理解你的app和UI。
  • 更好的可复用性。你可以在数据源完全不同的情况下使用同样的presentational组件，并将其分别放在一些以后会用到的container组件中。
  • presentational组件实际上是你的应用的“调色板”。你可以将他们放到一个单独的页面上，让设计者在不触及应用逻辑的情况下调整所有参数。你也可以在这个页面上运行回归测试。
  • 这迫使你提取出如Sidebar, Page, ContextMenu的“布局组件”，并使用this.props.children而不是将同样的标记和布局放在多个容器组件中。

记住，组件不必产生DOM。他们只需要提供UI关注点之间的组合边界。

好好利用这些优势。

什么时候引入容器组件？

我的建议是从建立一个只有展示组件的应用开始。最后你会意识到，你向中间的组件传递了太多的props。当你发现一些组件并不使用他们收到的props，而是单纯地将其向下传递，且当子组件需要的任何新数据你都需要重新将这些中间的组件串联起来的时候，应该是引入一些容器组件的好时机。这样你可以将数据和行为直接传递给组件树的叶子节点（组件），而不用修改组件树中的无关的组件。

这是一个逐步修改的过程，因此不要试图在一开始就做到。在你实践这个模式的时候，将会发展出一个何时抽取出一些容器的直观感觉，就像你知道什么时候提取出一个函数一样。我在Egghead上的Redux 系列视频也可能给你带来这方面的帮助！

其他的二分法

你必须要知道：展示组件和容器组件的区分并不是技术层面的。实际上，这是对他们意图的区分。

相比之下，这里有一些相关（但不同！）的技术层面的区分：

  • 有状态（Stateful）和无状态（Stateless）。有些组件会调用React的setState()方法而有些不用。虽然容器组件倾向于有状态，展示组件倾向于无状态，但这不是强制规定。展示组件可以是有状态的，容器也可以是无状态的。
  • 类和函数。从React 0.14开始，类能够以类或者函数的方式声明。函数组件在定义上更方便，但是缺少一些只有类组件能够使用的特性。这些限制可能会在以后去除，但是目前还算是存在的。因为函数形式的组件更容易理解，我建议你在不需要state，生命周期钩子（lifecircle hooks）或性能优化等当前只有类组件能用的特性时使用函数组件。
  • 纯和不纯。当一个组件在给定同样的props和state的时候一定饭hi一样的结果时，人们认为这个组件是“纯”的。“纯”组件可以通过类和函数的方式定义，也可以是有状态或无状态的。“纯”组件的另一个很重要的特点是他们不依赖props或state的深层变化，因此他们可以通过在他们的shouldComponentUpdate()钩子里使用浅层的比较来优化渲染性能。目前只有类能定义shouldComponentUpdate()，但未来可能会改变。

展示组件和容器组件可以落入这些划分出来的类别中。以我的经验来讲，展示组件倾向于无状态的纯函数，容器倾向于有状态的纯类。但这不是一个规则而只是一个意见，我见过在特定环境中讲得通的反例。

不要把展示组件和容器组件的区分当成一个教条。有些时候没必要或难以确定界限。如果你难以确定一个组件是展示组件的还是容器组件，你大可以晚点做决定。不要紧张！
